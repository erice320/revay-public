Contacts Endpoint
GET /company/contacts

All contacts across companies. Pull per company using conditions=company/id={id} AND inactiveFlag=false, or pull the full dataset and group by company.id in post-processing. MS Learn pyconnectwise schema

Extraction Logic

Filter communicationType="Email" where defaultFlag=true → primary email.

Filter communicationType="Phone" where defaultFlag=true → primary phone.

A contact can have multiple items of each type (work email, personal email, etc.) — defaultFlag identifies the primary one.

Libraries & Code Examples
Python
pyconnectwise
100% API coverage
pip install pyconnectwise from pyconnectwise import ConnectWiseManageAPIClient client = ConnectWiseManageAPIClient( company_name="acmeit", manage_url="api-na.myconnectwise.net", client_id="your-client-id", public_key="your-public-key", private_key="your-private-key" ) # Get all active companies companies = client.company.companies.get_all( params={ "conditions": 'status/name="Active" AND deletedFlag=false AND leadFlag=false' } ) # Get contacts per company for co in companies: contacts = client.company.contacts.get_all( params={"conditions": f"company/id={co.id} AND inactiveFlag=false"} ) # Get sites per company sites = client.company.companies.id(co.id).sites.get_all()

Customer Setup Steps

The migrating customer must complete these steps in their CW Manage instance before the migration tool can connect.

1

Log in as admin

Admin-level CW Manage account required to create API members

2

Create an API-Only Member

System → Setup Tables → Members → New → Type: API Member → Set username (e.g. revio-migration)

3

Assign Inquire permissions

Create a custom security role with Inquire access on: Companies/Company Maintenance · Companies/Contacts · Companies/Company Sites · Finance/Agreements (optional) · Company/Configurations (optional)

4

Generate API Keys

Members → select the API member → API Keys tab → Add Key → copy both Public Key and Private Key

5

Provide credentials to Rev.io

Company ID + Public Key + Private Key. Rev.io registers for a single Client ID at developer.connectwise.com and reuses it across all migrations.

Builder Gotchas

Hard-learned patterns from the CW API. Read before writing a single line of extraction code.

📧
No top-level email on Contact. The communicationItems[] array is the only source for email addresses and phone numbers. There is no root-level email or phone field on the Contact object — do not look for it.
🔑
Preserve identifier as the external key. company.identifier is the unique account code (e.g. "ACME-IT"). Store it in Rev.io as the external reference so future lookups can cross-reference the original CW record.
📍
Sites are a separate endpoint, not embedded. The Company object has a site reference (default site only). Full branch office list requires a separate call to /company/companies/{id}/sites.
🗑️
Always filter soft deletes. CW uses soft deletes — deletedFlag=true on Company and inactiveFlag=true on Contact. These records still appear in results unless explicitly filtered out.
📋
Include custom fields even if unmapped. userDefinedField1–10 exist on both Company and Contact. Include them in the export payload even if there's no Rev.io mapping yet — customers often have important data stored there.
🌎
Ask which region before connecting. NA, EU, and AU instances use different base URLs. On-prem customers have a custom domain entirely. Confirm upfront — a wrong base URL gives a misleading auth error.
